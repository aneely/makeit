# Session Notes — 2026-02-23

## What Happened

The session started with a previous session's auto-generated code being wiped and restarted from scratch. The previous session had rushed ahead and built everything without adequate user input. This session was largely spent diagnosing why that happened and building the right conditions for collaborative work going forward.

The second half of the session built `bin/makeit` and the full BATS test suite collaboratively and iteratively — one piece at a time with user input at each step. That part worked well.

## Key Learnings

### Aspirational norms don't produce behavior — explicit constraints do
The original AGENTS.md described good working style but didn't prevent the model from filling ambiguity with forward movement. The fix was rewriting the collaboration norms as direct commands with explicit stops: "one move, then stop, then wait for response — no exceptions."

### The planning phase and the building phase have different dynamics
Planning enforces turns naturally — every decision requires input before there's anything to build. Building doesn't. Without explicit constraints, the building phase becomes a greased path toward a finished artifact the user had no input on.

### Behavioral language from boiler's PROJECT-CONTEXT.md is more effective
Boiler prohibits bad behavior by name ("Do NOT automatically implement", "STRICTLY provide explanation only"). Makeit's original norms described good behavior aspirationally. Prohibition is more effective than aspiration.

### Completing a task without user input is a failure, not an accomplishment
The measure of a good session is whether the user was involved in every decision — not whether code was produced. A finished artifact the user didn't shape is the wrong outcome.

### Rolling horizon > full backlog
Upfront story decomposition is waterfall with extra steps. The right approach: agree on the next thing to build and why, build it together, update the story file at the end. Keep a few items of lookahead — enough to know what's next, not enough to require upfront design of things not yet reached.

### Human-in-the-loop value is design input, not end approval
The human brings priorities, taste, and knowledge of tradeoffs the model doesn't have: where to make a hard change easy, when a procedure should be a composable function, when verbosity is a smell. That input only works if it happens during the build, not after.

## What's Working

- One move per exchange, stop and wait
- Proposing test cases before writing tests, then writing tests before implementation
- Reviewing files together before deciding anything about them
- The kanban story file as source of truth with a markdown checklist
- Commit messages scoped to what was actually agreed and built

## Discoveries

### Onboarding tension: default path discourages version control
The default `CONFIG_DIR` (`~/.config/makeit`) minimizes new-user friction but actively
works against the portability and version-control values the project is built around.
Real users (including the project author) will want profiles in `~/dev/` under git.
Captured as `plans/todo/onboarding-config-repo.md` — next story to tackle.

### PATH/install step missing from new-user flow
Manual test surfaced `makeit: command not found` — `make install` hasn't been run, so
`~/.local/bin` isn't on PATH. The README new-user flow needs to address this explicitly.
Noted in the onboarding story.

### LDoc for agent-friendly codebase summaries
Discussion about structured comment standards (LDoc for Lua) as a way to give agent
sessions a comprehensive view of the codebase at low token cost. Captured as
`plans/todo/lua-doc-comments.md`.

## What's Working

- `DEFAULT_CONFIG_DIR` pattern for Makefile test overrides — clean, no real-dir side effects
- Keeping story files current as work progresses (not just at the end)
- Capturing discoveries as story stubs mid-session rather than losing them
